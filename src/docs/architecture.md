============================================================
                  معماری پروژه (Architecture)
============================================================

نام پروژه: my_bot_project
معماری: Clean Architecture + Domain-Driven Design (DDD)
نسخه: 1.0.0

------------------------------------------------------------
                    نمای کلی معماری
------------------------------------------------------------

این پروژه بر اساس اصول Clean Architecture و Domain-Driven Design
(DDD) طراحی شده است. هدف اصلی، جداسازی کامل لایه‌ها و کاهش
وابستگی‌ها به گونه‌ای است که تغییرات در هر لایه، تأثیر حداقلی
بر سایر لایه‌ها داشته باشد.

معماری پروژه به ۴ لایه اصلی تقسیم می‌شود که هر کدام مسئولیت
مشخصی دارند و وابستگی‌ها از بیرون به داخل است (لایه‌های داخلی
به لایه‌های خارجی وابسته نیستند).

                       +-------------------+
                       |   Presentation    |  (لایه ارائه)
                       +-------------------+
                               |
                       +-------------------+
                       |   Application     |  (لایه کاربرد)
                       +-------------------+
                               |
                       +-------------------+
                       |     Domain        |  (لایه دامنه)
                       +-------------------+
                               |
                       +-------------------+
                       |  Infrastructure   |  (لایه زیرساخت)
                       +-------------------+

------------------------------------------------------------
                    لایه Domain (دامنه)
------------------------------------------------------------

لایه Domain هسته اصلی پروژه است و هیچ وابستگی به لایه‌های دیگر
ندارد. این لایه شامل موارد زیر است:

۱. Entities (موجودیت‌ها):
   موجودیت‌های اصلی کسب‌وکار که دارای هویت منحصر‌به‌فرد هستند.
   
   - User (کاربر)
   - Order (سفارش)
   - Payment (پرداخت)
   - Coupon (کوپن تخفیف)
   - Form (فرم)
   - FormResponse (پاسخ فرم)
   - Ticket (تیکت پشتیبانی)
   - Broadcast (پیام گروهی)
   - Feedback (بازخورد)
   - ABTest (تست A/B)
   - AuditLog (لاگ عملیات)

۲. Value Objects (ارزش‌مقادیر):
   اشیایی که هویت ندارند و صرفاً برای نمایش مقدار استفاده می‌شوند.
   
   - Email (ایمیل)
   - Phone (شماره تلفن)
   - Money (مبلغ)
   - RateLimit (محدودیت نرخ)
   - FormField (فیلد فرم)
   - UserLevel (سطح کاربر)

۳. Interfaces (اینترفیس‌ها):
   قراردادهایی که لایه‌های دیگر باید پیاده‌سازی کنند.
   
   - UserRepository
   - OrderRepository
   - PaymentRepository
   - CouponRepository
   - FormRepository
   - TicketRepository
   - CacheInterface
   - MessagePublisher

۴. Enums و Constants:
   مقادیر ثابت و شمارشی مورد استفاده در دامنه.

   - OrderStatus (وضعیت سفارش)
   - PaymentStatus (وضعیت پرداخت)
   - UserRole (نقش کاربر)
   - FormType (نوع فرم)

------------------------------------------------------------
                    لایه Application (کاربرد)
------------------------------------------------------------

لایه Application منطق کسب‌وکار را پیاده‌سازی می‌کند و از
اینترفیس‌های لایه Domain استفاده می‌کند. این لایه شامل موارد زیر است:

۱. Services (سرویس‌ها):
   پیاده‌سازی Use Cases و منطق اصلی برنامه.
   
   - UserRegistrationService (ثبت‌نام کاربر)
   - UserProfileService (پروفایل کاربر)
   - OrderCreationService (ایجاد سفارش)
   - OrderStatusUpdateService (بروزرسانی وضعیت سفارش)
   - PaymentGatewayService (درگاه پرداخت)
   - PaymentVerificationService (تأیید پرداخت)
   - FormBuilderService (ساخت فرم)
   - FormSubmissionService (ثبت فرم)
   - BroadcastSenderService (ارسال گروهی)
   - BroadcastFilterService (فیلتر ارسال گروهی)
   - TicketService (مدیریت تیکت)
   - CouponService (مدیریت کوپن)

۲. DTOs (Data Transfer Objects):
   اشیایی برای انتقال داده بین لایه‌ها.
   
   - UserDTO
   - OrderDTO
   - PaymentDTO
   - CouponDTO
   - FormDTO
   - BroadcastDTO

۳. Use Cases:
   کیس‌های استفاده مشخص و مستقل.
   
   - RegisterUser (ثبت‌نام کاربر)
   - UpdateProfile (بروزرسانی پروفایل)
   - CreateOrder (ایجاد سفارش)
   - CancelOrder (لغو سفارش)
   - InitiatePayment (شروع پرداخت)
   - ConfirmPayment (تأیید پرداخت)

------------------------------------------------------------
                    لایه Infrastructure (زیرساخت)
------------------------------------------------------------

لایه Infrastructure پیاده‌سازی‌های عینی اینترفیس‌های لایه Domain
را فراهم می‌کند. این لایه شامل موارد زیر است:

۱. Database (دیتابیس):
   اتصال به دیتابیس و مدیریت Session.
   
   - SessionManager (مدیریت جلسات)
   - ConnectionPool (مدیریت اتصالات)
   - Models (مدل‌های SQLAlchemy)
   - Migrations (مهاجرت‌های Alembic)

۲. Repositories (ریپازیتوری‌ها):
   پیاده‌سازی اینترفیس‌های Repository.
   
   - UserRepoImpl
   - OrderRepoImpl
   - PaymentRepoImpl
   - CouponRepoImpl
   - FormRepoImpl
   - TicketRepoImpl

۳. Cache (کش):
   پیاده‌سازی کش با Redis و Fallback محلی.
   
   - CacheManager (مدیریت کش)
   - RedisAdapter (اتصال به Redis)
   - LocalAdapter (کش محلی)
   - CacheFallback (Fallback خودکار)

۴. External Services (سرویس‌های خارجی):
   ارتباط با سرویس‌های خارجی.
   
   - PaymentGateway (درگاه پرداخت)
   - EmailSender (ارسال ایمیل)
   - SmsSender (ارسال پیامک)

۵. Health Check (بررسی سلامت):
   بررسی سلامت سرویس‌ها.
   
   - HealthChecker
   - DBHealth
   - CacheHealth
   - ExternalHealth

------------------------------------------------------------
                    لایه Presentation (ارائه)
------------------------------------------------------------

لایه Presentation ورودی کاربر را دریافت کرده، به لایه Application
ارسال می‌کند و پاسخ را به کاربر نمایش می‌دهد. این لایه شامل
موارد زیر است:

۱. Handlers (هندلرها):
   پردازش پیام‌ها و Callback Query‌ها.
   
   - StartHandler
   - ProfileHandler
   - FormHandlers
   - PaymentHandlers
   - BroadcastHandlers
   - AdminHandlers

۲. Keyboards (کیبوردها):
   تولید دکمه‌های Inline و Reply Keyboard.
   
   - CommonKeyboards (دکمه‌های مشترک)
   - UserKeyboards (دکمه‌های کاربر)
   - AdminKeyboards (دکمه‌های مدیریت)

۳. Middlewares (میدلورها):
   پردازش‌های میانی قبل از رسیدن به Handler.
   
   - RateLimiter (محدودیت نرخ درخواست)
   - LoggingMiddleware (لاگ‌گیری)
   - I18nMiddleware (چندزبانی)
   - FeatureFlagMiddleware (فیچر فلاگ)

۴. Web API (وب‌هوک):
   آمادگی برای ارتباط با وب‌هوک.
   
   - WebApp (اپلیکیشن وب)
   - WebhookRoutes (مسیرهای وب‌هوک)
   - HealthRoutes (مسیرهای سلامت)

                    جریان داده (Data Flow)
------------------------------------------------------------

۱. کاربر پیامی ارسال می‌کند یا روی دکمه‌ای کلیک می‌کند.
۲. لایه Presentation پیام را دریافت کرده و به Middleware می‌فرستد.
۳. Middlewareها پردازش‌های اولیه (مثل Rate Limiting) را انجام می‌دهند.
۴. Handler مناسب، درخواست را پردازش کرده و سرویس مربوطه را صدا می‌زند.
۵. لایه Application منطق کسب‌وکار را اجرا کرده و با لایه Domain
   و Infrastructure ارتباط برقرار می‌کند.
۶. لایه Infrastructure عملیات دیتابیس، کش و سرویس‌های خارجی را انجام می‌دهد.
۷. نتیجه به Handler برگردانده می‌شود.
۸. Handler پاسخ را به صورت پیام یا کیبورد به کاربر نمایش می‌دهد.

------------------------------------------------------------
                    Dependency Injection (DI)
------------------------------------------------------------

پروژه از یک کانتینر DI ساده برای مدیریت وابستگی‌ها استفاده می‌کند.
همه سرویس‌ها، ریپازیتوری‌ها و سایر اجزاء در Container ثبت می‌شوند.

مثال ثبت سرویس:

class Container:
    def __init__(self, config: Config):
        self.config = config
        self.db_manager = DatabaseSessionManager(config.db_url)
        self.user_repo = UserRepoImpl(self.db_manager)
        self.cache = CacheManager(config.redis_url)
        self.user_service = UserService(self.user_repo, self.cache)

مزایای استفاده از DI:
- کاهش وابستگی‌های سخت
- افزایش قابلیت تست‌پذیری
- مدیریت آسان‌تر وابستگی‌ها
- امکان جایگزینی پیاده‌سازی‌ها

پروژه از یک سلسلهمراتب استثناهای سفارشی استفاده می‌کند:

MyBotError (کلاس پایه)
├── ConfigurationError
├── DatabaseError
├── CacheError
├── ValidationError
├── PermissionDeniedError
├── RateLimitExceededError
├── FormProcessingError
├── BroadcastError
├── FeatureDisabledError
└── NotFoundError

همه استثناها در core/exceptions.py تعریف شده‌اند و در سراسر
پروژه استفاده می‌شوند.

                    لاگ‌گیری (Logging)
------------------------------------------------------------

لاگ‌گیری با استفاده از RotatingFileHandler پیاده‌سازی شده است:

- سطح لاگ: DEBUG (توسعه) / INFO (تولید)
- فرمت: تاریخ - نام - سطح - پیام
- چرخش فایل: پس از رسیدن به ۱۰ مگابایت
- تعداد فایل‌های پشتیبان: ۵

سطوح لاگ:
- ERROR: خطاهای بحرانی
- WARNING: هشدارها
- INFO: اطلاعات عمومی
- DEBUG: اطلاعات دقیق برای عیب‌یابی

------------------------------------------------------------
                    کش (Cache)
------------------------------------------------------------

سیستم کش با قابلیت Fallback خودکار طراحی شده است:

۱. Redis (اولویت اول)
۲. Local (Fallback در صورت عدم دسترسی به Redis)

مکانیزم Fallback:
- اگر Redis در دسترس نباشد، به‌صورت خودکار به Local Cache سوئیچ می‌شود.
- در صورت بازیابی Redis، سیستم دوباره به Redis برمی‌گردد.

پارامترهای کش:
- TTL پیش‌فرض: ۳۰۰ ثانیه
- کلیدها: با prefix مشخص (مثلاً user:123)

------------------------------------------------------------
                    امنیت (Security)
------------------------------------------------------------

۱. احراز هویت:
   - کاربران ادمین در دیتابیس ذخیره می‌شوند.
   - تنها ادمین‌ها به پنل مدیریت دسترسی دارند.

۲. مجوزها (Permissions):
   - کنترل دسترسی بر اساس نقش (Role-Based Access Control)
   - نقش‌ها: super_admin, admin, moderator, support

۳. محدودیت نرخ درخواست (Rate Limiting):
   - پیاده‌سازی با الگوریتم Sliding Window
   - پیش‌فرض: ۳۰ درخواست در ۶۰ ثانیه

۴. Feature Flag:
   - امکان فعال/غیرفعال کردن ویژگی‌ها به‌صورت پویا

۵. اعتبارسنجی ورودی:
   - استفاده از Pydantic برای اعتبارسنجی تمام ورودی‌ها

------------------------------------------------------------
                    نکات کلیدی طراحی
------------------------------------------------------------

۱. Single Responsibility Principle (SRP):
   - هر کلاس فقط یک مسئولیت دارد.

۲. Dependency Inversion Principle (DIP):
   - لایه‌های بالا به اینترفیس‌ها وابسته‌اند، نه پیاده‌سازی.

۳. Open/Closed Principle (OCP):
   - کلاس‌ها برای توسعه باز و برای تغییر بسته هستند.

۴. Interface Segregation Principle (ISP):
   - اینترفیس‌ها کوچک و متمرکز هستند.

۵. Liskov Substitution Principle (LSP):
   - کلاس‌های مشتق‌شده قابل جایگزینی با کلاس پایه هستند.

۶. Separation of Concerns:
   - جداسازی کامل لایه‌های مختلف.

۷. Testability:
   - طراحی به‌گونه‌ای که تست‌نویسی آسان باشد.
